### 3️⃣ Programmation PL/pgSQL (`03-programmation.sql`)

Ce script contient toute la logique métier : procédures, fonctions et triggers.

#### 🔹 Procédure `ajouter_client`

```sql
CREATE OR REPLACE PROCEDURE ajouter_client(
    p_nom TEXT, 
    p_prenom TEXT, 
    p_telephone TEXT, 
    p_email TEXT
)
LANGUAGE plpgsql
AS $$
BEGIN
    -- Validation : nom et prénom obligatoires
    IF p_nom IS NULL OR p_prenom IS NULL THEN
        RAISE EXCEPTION 'Nom et prénom sont obligatoires';
    END IF;
    
    -- Validation : format email
    IF p_email !~ '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$' THEN
        RAISE EXCEPTION 'Format email invalide';
    END IF;
    
    -- Vérification : email unique
    IF EXISTS (SELECT 1 FROM exchange.client WHERE email = p_email) THEN
        RAISE EXCEPTION 'Email déjà utilisé : %', p_email;
    END IF;
    
    INSERT INTO exchange.client(nom, prenom, telephone, email)
    VALUES (p_nom, p_prenom, p_telephone, p_email);
    
    RAISE NOTICE 'Client ajouté avec succès : % %', p_prenom, p_nom;
END;
$$;
```

## ⚙️ Fonctionnalités implémentées

### 🔹 1. Procédure `ajouter_client`

Permet d'ajouter un nouveau client avec validation complète :

| Validation | Description |
|------------|-------------|
| ✅ Nom & Prénom | Champs obligatoires |
| ✅ Email | Format validé par expression régulière |
| ✅ Unicité | Vérification email unique |

📌 **Exemple d'utilisation :**
```sql
CALL exchange.ajouter_client('Nemous', 'Abdelatif', '6471230000', 'abdelatif.nemous@gmail.com');
```

---

### 🔹 2. Fonction `nombre_transactions_client`

Retourne le nombre de transactions pour un client donné.

📌 **Exemple :**
```sql
SELECT exchange.nombre_transactions_client(1);
```

---

### 🔹 3. Procédure `ajouter_transaction`

Permet d'ajouter une transaction avec :

- ✅ Validation du client (existence)
- ✅ Validation des devises (source et cible)
- ✅ Vérification du taux de change
- ✅ Calcul automatique du montant converti
- ✅ Insertion dans l'historique

---

### 🔹 4. Trigger `trg_valider_transaction`

| Propriété | Valeur |
|-----------|--------|
| **Type** | `BEFORE INSERT / UPDATE` |
| **Fonction** | Validation automatique des données |

**Validations effectuées :**
- Montant initial > 0
- Montant converti > 0
- Devise source ≠ devise cible

---

### 🔹 5. Trigger `trg_log_transaction`

| Propriété | Valeur |
|-----------|--------|
| **Type** | `AFTER INSERT / UPDATE / DELETE` |
| **Fonction** | Journalisation automatique |

Ajoute automatiquement une entrée dans `exchange.historique_transaction` à chaque modification.

## 🧠 Conclusion

Ce TP démontre l'importance de **centraliser la logique métier directement dans la base de données** à l'aide de PL/pgSQL.

### Avantages constatés :

| Avantage | Description |
|----------|-------------|
| 🔒 **Intégrité des données** | Les contraintes et triggers garantissent la cohérence |
| ⚡ **Performance** | Traitement côté serveur, réduction des allers-retours |
| 🛡️ **Sécurité** | Validation centralisée, impossible de contourner |
| 📊 **Audit** | Journalisation automatique de toutes les actions |

Les **triggers** permettent une automatisation efficace, tandis que les **procédures et fonctions** assurent la cohérence et la validation des données.
